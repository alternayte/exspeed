# Exspeed Benchmarks

_This file is regenerated by `exspeed-bench render`. Do not edit manually._

_Status: reference numbers pending the first Hetzner CCX33 refresh run. Until then, this document describes the methodology; use `exspeed-bench all --profile local` to generate your own numbers locally._

## How these numbers were produced

- **Host (reference):** Hetzner CCX33 — 8 dedicated vCPU (AMD EPYC), 32 GB RAM, 240 GB NVMe, 10 Gbit/s NIC. Ubuntu 24.04 LTS.
- **Broker + workload driver on the same box.** Single-node measurement.
- **Workload driver:** `exspeed-bench`, a Rust binary that speaks the wire protocol directly via the `exspeed-protocol` crate. Latency is recorded in hdrhistograms with coordinated-omission correction.
- **Every publish carries a `bench.publish_us` header** with a shared-origin microsecond timestamp; the consumer computes the delta on the same machine so there is no clock-skew risk.

## How to reproduce

```bash
git checkout <sha>
cargo build --release -p exspeed-bench
./target/release/exspeed -- server --data-dir /var/tmp/exspeed-bench &
./target/release/exspeed-bench all --profile reference \
  --sku hetzner-ccx33 --storage nvme-ext4 \
  --output bench/results/refresh.json
./target/release/exspeed-bench render bench/results/refresh.json --out BENCHMARKS.md
```

## Scenarios

### Publish

Measures single-producer ceiling in msg/s and MB/s across three payload sizes (100 B, 1 KB, 10 KB). Four concurrent producer tasks on one stream, 60 s per payload size, unlimited rate.

### Latency

Measures end-to-end tail latency at a sustained 100k msg/s input rate with a 1 KB payload. Reports p50, p90, p99, p99.9, p99.99 and max from an hdrhistogram. 60 s.

### Fan-out

Measures read-side scaling. One producer at 50k msg/s feeds a single stream; we run with 1, 4, 16, and 64 concurrent consumers on that stream and report aggregate consumer throughput plus per-consumer lag.

### ExQL

Runs a continuous aggregation query (`SELECT subject, COUNT(*) FROM bench GROUP BY subject EMIT CHANGES`) while publishing to the source stream, and binary-searches the highest rate the producer sustains (producer at ≥95% of target for the full window). This measures the broker write path WITH an active continuous query running — not the query-apply ceiling itself. Binary-search: 6 iterations, bounds 5k–500k msg/s. A follow-up plan will add a direct apply-lag measurement that queries the output stream's offset vs source offset.

ExQL aggregation query over a single-subject stream (`subject = "bench"` throughout).
